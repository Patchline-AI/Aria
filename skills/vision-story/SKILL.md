## Your Task

Produce `.patchline/artifacts/VISION.md`, a concise document that captures the project's sonic identity, narrative positioning, and tonal palette. Every factual claim must trace to BRIEF.md, AUDIO_INTAKE.md, MCP output, or the user's words.

Do not ask for peer artists by default. Use existing similar/peer data from artist intelligence or track analysis when available; otherwise skip peer references. If the user volunteers a reference artist, validate it with `mcp__aria__search_artists`.

## Step 1: Read Workspace Context

Use Read in this order:

- `.patchline/STATE.md` - confirm `Current phase: vision-story`; extract `Composition status`, `Focus track asset ID`, `Audio status`, and `Track analysis status` (tolerate legacy `cynite_*` status values if present).
- `.patchline/PROJECT.md` - extract artist name, project title, genres, career stage, distribution mode, Project Anchor ID, and campaign intake.
- `.patchline/artifacts/AUDIO_INTAKE.md` - if present, extract focus-track title, asset ID, track-analysis summary, and campaign intake.
- `.patchline/artifacts/BRIEF.md` - extract project intent, audience, success metric, and non-negotiables.
- `.patchline/artifacts/VISION.md` - if it already exists, ask whether to overwrite, refine, or cancel.

If BRIEF.md is missing, stop and tell the user to run `/aria:next`; it will detect drift and route back.

## Step 2: Ground In Data

If a focus track exists, call:

- `mcp__aria__get_asset` with the focus-track asset ID.
- `mcp__aria__get_audio_features` with the same asset ID.

Capture BPM, key, genres, moods, energy, danceability, valence, acousticness, and any generated audio description if present.

If artist genre is known, optionally call `mcp__aria__get_trending_artists` to understand current peer landscape. Use it only as context; do not force the artist into that lane.

If MCP tools error, note the exact error and continue with the artifacts on disk. Do not fabricate peer lists or audio facts.

## Step 3: Interview Lightly

Ask questions one at a time. Finished-single projects should usually need only 2-3 questions because AUDIO_INTAKE.md and BRIEF.md already carry the heavy context.

1. If track analysis exists, ask: "I will use the track read as the sonic base: <BPM/key/genre/mood summary>. If that misses something important, correct it in one sentence; otherwise say `use it`." If no focus track exists, ask for a 2-3 sentence sound description.
2. Ask: "What is the one-line story behind this project? Not marketing copy - the real why." Skip if BRIEF.md already contains it.
3. Ask: "Give me three words for the emotional register. Not the genre - the feeling." Skip if BRIEF.md/campaign intake already has clear tonal words.

Do not ask "Name 1-3 artists" by default. If the user volunteers references in any answer, validate them in Step 4.

## Step 4: Validate Volunteered References Only

For each user-volunteered reference artist, call `mcp__aria__search_artists`.

- Exact match - use canonical name.
- Fuzzy match - ask whether that is what they meant.
- No match - ask whether to drop it or keep it flagged as unverified.

If no references were volunteered, write "No external reference artists selected; downstream phases will use the focus track and catalog references instead."

## Step 5: Draft VISION.md

Use Write to create `.patchline/artifacts/VISION.md`:

```markdown
# Vision: <project-name>

> Generated by `/aria:vision-story` on YYYY-MM-DD - v1
> Grounded in: BRIEF.md, AUDIO_INTAKE.md / `get_asset` / `get_audio_features` if a focus track exists, optional peer-landscape check, user interview.

## Sonic identity

<2-3 sentences grounded in track analysis plus the user's one-sentence correction, if any. If the user said "use it", trust the track read.>

## Narrative / story

<One tight paragraph from the user's story, or from BRIEF.md if already captured.>

## Tonal palette

- **<word 1>** - <what it means for this project>
- **<word 2>** - <what it means>
- **<word 3>** - <what it means>

## Reference posture

<Confirmed volunteered references, unverified references, or "No external reference artists selected; focus track and catalog references carry the moodboard.">

## Peer-landscape context

<If a peer tool returned useful data, summarize in 2-3 sentences. If not, write "Peer-landscape check unavailable or skipped; no peers invented.">

## Data sources

- BRIEF.md: <date if available>
- Focus-track audio: <asset ID + summary or unavailable>
- Artist/peer context: <tool used or skipped>
- User interview: <date, N questions answered>
```

## Step 6: Update STATE.md

Append only:

- `- vision-story - completed YYYY-MM-DD, artifact: VISION.md` to `Completed phases:`.
- `- VISION.md (vision-story phase, generated YYYY-MM-DD)` to `Artifacts:`.
- Update `Last updated: YYYY-MM-DD by /aria:vision-story`.

Do not touch `Current phase`.

## Step 7: Hand Off

Tell the user in three sentences or fewer that VISION.md is written, name the sonic base and story in one phrase, and ask them to run `/aria:next` to move into moodboard.

## Error Handling

- If track-analysis tools fail but AUDIO_INTAKE.md has a summary, use the cached artifact and mark live lookup unavailable.
- If the user says `use it`, do not ask a second sonic-correction question.
- If peer/trending tools return empty results, skip peer context; no invented references.
- If BRIEF.md is missing, stop and route through `/aria:next` instead of drafting VISION.md out of order.

## Examples

Finished single: "I will use the track read as the sonic base: 126 BPM, D minor, atmospheric dance track. Correct it in one sentence, or say `use it`." Then write VISION.md from that read plus BRIEF.md.

Idea-first project: ask for a short sound description because no focus-track analysis exists, then keep reference artists optional and user-volunteered only.

## Common Mistakes

- Asking for peer artists by default.
- Asking the user to describe the sound from scratch when track analysis exists.
- Inventing peer names or trend context.
- Turning "idk" into fake marketing language. If the user is terse, keep the artifact honest and thin.
